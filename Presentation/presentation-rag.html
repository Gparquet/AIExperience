Les embeddings : GPS du sens

Comment l'IA retrouve des documents par le sens — pas par les mots-clés

🗺️ L'analogie de la carte

Imaginez une carte où chaque texte est placé selon son sens. Deux textes qui parlent du même sujet sont proches sur la carte, même s'ils utilisent des mots complètement différents.

📍 Exemple concret
🟢 "Comment poser des congés ?"
→ proche de "Politique de RTT", "Vacances annuelles"
🔵 "Frais de transport ?"
→ proche de "Note de frais", "Remboursement"
💡 Pourquoi mieux que Ctrl+F ?
Un Ctrl+F cherche les mots exacts. L'embedding comprend que « congés payés » et « vacances » ont le même sens et retourne le bon document.

Le Chunking — 40% des problèmes RAG

Le mauvais découpage est la première cause d'échec d'un système RAG

⚠️ Le problème : similarité ≠ pertinence

Question sur « politique de conservation des données » → le RAG naïf retourne des résultats sur les « programmes de fidélisation » car le mot « conservation » est partagé.

💡 Comment ça marche : après chaque chunk, le curseur avance de size − overlap au lieu de size.
pos += size − overlap  // 800 − 100 = 700
Règle empirique : 10–25% de la taille du chunk selon le type de document.
⚠️ Les deux erreurs classiques
Trop petit < 5%phrases coupées non récupérées — réponse tronquée malgré l'info présente
Trop grand > 30%doublons en top-K — signal dilué, tokens gaspillés

Stratégies de récupération avancées

Selon la complexité de la question, différentes stratégies s'appliquent

⚡ Stratégie Direct

La plus simple. La question est directement transformée en vecteur et on recherche les chunks les plus proches.

Quand : Questions courtes et factuelles
Exemple : "Quelle est la date de la réunion X ?"
Avantage : rapide, déterministe
Limite : fragile sur questions complexes
💡 HyDE — Hypothetical Document Embedding

Le LLM génère d'abord un document hypothétique qui ressemble à la réponse attendue. L'embedding de ce doc — plutôt que de la question — est bien plus proche des vrais chunks.

Quand : questions ouvertes, complexes, inférences
Exemple : "Expliquez les principes du GBCP"
Avantage : vecteur de recherche sémantiquement plus riche
🔀 RAG-Fusion — Multi-Query + RRF

Le LLM génère 3 reformulations de la question. Chaque reformulation fait une recherche séparée. Les résultats sont fusionnés par RRF (Reciprocal Rank Fusion) : score = Σ 1/(k+rang)

Quand : comparaisons, multi-aspects, synthèse
Exemple : "Comparez le GBCP et le Code des marchés"
Avantage : couverture maximale, réduit les angles morts
🧠 Adaptive — Routeur intelligent

Le LLM analyse la question et choisit automatiquement la stratégie optimale. C'est le mode par défaut en production — transparent pour l'utilisateur.

// AdaptiveQueryRouter.cs if (raw.Contains("HyDE")) return RagStrategy.HyDE; if (raw.Contains("Fusion")) return RagStrategy.Fusion; return RagStrategy.Direct; // fallback sécurisé

Reranking & Compression

Réduire le bruit progressivement — qualité > quantité

⚠️ Attention au "Context Stuffing" : injecter 20 chunks "au cas où" quand seuls 3 sont pertinents dilue le signal. Le LLM ne trie pas — il moyenne tout. Plus de chunks ≠ meilleure réponse.

-- Schéma pgvector — extension cosinus CREATE EXTENSION IF NOT EXISTS vector; CREATE TABLE document_chunks ( id UUID PRIMARY KEY, document_id UUID REFERENCES documents(id), content TEXT NOT NULL, embedding vector(768), -- nomic-embed-text chunk_index INT, page_number INT ); -- Index HNSW : O(log n) vs O(n) CREATE INDEX ON document_chunks USING hnsw (embedding vector_cosine_ops) WITH (m = 16, ef_construction = 64); -- Recherche Top-K par similarité cosinus SELECT content, 1-(embedding <=> $1::vector) AS score FROM document_chunks ORDER BY score DESC LIMIT 10;

RAG vs alternatives

Quand utiliser quoi ? Les trois approches sont complémentaires.

Critère 🔍 RAG 🧠 Fine-tuning 📋 Contexte direct
Mise à jour des données✅ Instantanée (ajouter des docs)❌ Réentraînement nécessaire✅ Modifier le fichier
Volume documentaire✅ Milliers de documents⚠️ Données d'entraînement❌ Limité à ~20 docs
Sources vérifiables✅ Citations précises❌ Opaque⚠️ Partiel
Coût✅ Faible (indexation 1×)❌ Très élevé✅ Gratuit
Données privées✅ On-premise possible⚠️ Dépend du provider✅ Local
Questions imprévues✅ Recherche sémantique auto✅ Si dans les données❌ Règles manuelles
Précision sur niche⚠️ Dépend du chunking✅ Excellente⚠️ Si doc disponible
Recommandé pourBases documentaires largesStyle / domaine très spécifiqueRègles stables, <20 docs clés
💡 En pratique : utilisez le contexte direct pour vos règles métier stables (10-20 docs), le RAG pour vos grandes bases documentaires, et le fine-tuning uniquement si vous avez un besoin de style ou de domaine très spécifique non couvert par les deux premiers.

Points clés à retenir

Ce qu'il faut garder en tête pour construire un RAG robuste

✂️
Le chunking est la clé — 40% des problèmes RAG viennent d'un mauvais découpage. Overlap obligatoire.
🔍
Debug retrieval first — ~80% des échecs viennent de la récupération, pas du modèle. Inspectez les chunks avant de changer de LLM.
💡
HyDE & RAG-Fusion surpassent le RAG direct sur les questions complexes et comparatives.
🏆
Reranking (CrossEncoder) améliore significativement la précision finale — qualité > quantité.
🗄️
pgvector + PostgreSQL — un seul backend, EF Core natif, transactions ACID. Idéal pour démarrer.
🎥
Vidéo & Audio s'ingèrent comme des PDFs via FFmpeg + Whisper 100% local. Aucun coût cloud.
📊
Mesurez : faithfulness > 85%, context recall > 80%. Sans métriques, on ne sait pas si les améliorations fonctionnent.
🔒
Vos données restent chez vous — LM Studio local, Whisper local, pgvector on-premise. Zéro envoi cloud obligatoire.
